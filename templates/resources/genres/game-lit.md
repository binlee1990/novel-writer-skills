# 游戏/系统文 (GameLit / LitRPG)

## 快速参考

### 核心定义

**游戏文/系统文**：以游戏化机制（等级、技能、装备、副本、任务）为核心叙事框架的小说类型。读者享受的是「数值成长的确定性快感」和「系统化的世界规则」。

**与传统玄幻的区别**：
- 玄幻：力量体系模糊，作者自由度高
- 游戏文：力量体系数值化，规则明确，读者可以「算账」

### 子类型速查

| 子类型 | 核心特征 | 典型设定 |
|--------|---------|---------|
| LitRPG | 完整的 RPG 数值系统 | 等级、属性面板、技能树 |
| 系统流 | 主角独有系统 | 签到系统、抽奖系统、任务系统 |
| 副本流 | 以副本挑战为主线 | 无限流、副本世界、试炼塔 |
| 游戏世界 | 角色在游戏世界中冒险 | VRMMO、穿越到游戏世界 |
| 数据化 | 世界本身数据化 | 全球数据化、万物皆可升级 |

## 核心系统设计

### 1. 属性面板

**设计原则**：
- 属性数量 5-8 个（太少无趣，太多复杂）
- 每个属性必须有明确的游戏效果
- 属性之间应有取舍关系（不能全部堆满）

**通用属性模板**：
| 属性 | 效果 | 叙事价值 |
|------|------|---------|
| 力量 | 物理伤害、负重 | 战斗描写的基础 |
| 敏捷 | 速度、闪避 | 战斗节奏控制 |
| 体质 | 生命值、防御 | 生存能力 |
| 智力 | 魔法伤害、技能效果 | 策略性战斗 |
| 精神 | 魔力值、抗性 | 持久战能力 |
| 幸运 | 暴击、掉落 | 随机事件、惊喜 |

**属性展示频率**：
- 每次升级时展示完整面板
- 战斗中只提及关键属性变化
- 不要每章都展示完整面板（读者会跳过）

### 2. 技能系统

**技能分类**：
| 类型 | 获取方式 | 成长方式 |
|------|---------|---------|
| 主动技能 | 学习/领悟/系统奖励 | 使用次数/等级提升 |
| 被动技能 | 天赋/装备/觉醒 | 条件触发/等级提升 |
| 终极技能 | 特殊条件解锁 | 固定/不可升级 |

**技能设计要点**：
- 每个技能必须有明确的使用场景
- 技能之间应有组合效果（combo）
- 高级技能应有使用限制（冷却/消耗/条件）
- 技能描述要简洁，效果要具象

**技能展示原则**：
- 新技能获得时详细描述
- 战斗中用效果描写代替面板展示
- 避免「技能名 + 伤害数字」的机械式描写

### 3. 装备系统

**品质分级**（常用）：
```
白色（普通）→ 绿色（优秀）→ 蓝色（精良）→ 紫色（史诗）→ 橙色（传说）→ 红色（神话）
```

**装备叙事价值**：
- 装备获取 = 奖励感（爽点）
- 装备对比 = 成长感（数值提升可视化）
- 装备选择 = 策略感（取舍决策）
- 套装效果 = 收集感（长期目标）

### 4. 副本设计

**副本结构模板**：
```
入口/传送 → 环境描写 → 小怪区 → 谜题/机关 → 精英怪 → Boss 前置 → Boss 战 → 奖励结算
```

**副本设计要点**：
| 要素 | 说明 |
|------|------|
| 难度曲线 | 前松后紧，Boss 战是最难的部分 |
| 环境叙事 | 副本环境本身讲述故事（废墟、遗迹、历史） |
| 机制设计 | 每个 Boss 有独特机制，不能只是「血厚伤害高」 |
| 团队配合 | 如果是组队副本，每个角色都要有作用 |
| 意外元素 | 隐藏房间、隐藏 Boss、随机事件 |

## 数值化叙事技巧

### 数值展示的度

**核心问题**：数值展示太多 = 读者跳过；太少 = 失去游戏文特色。

**建议频率**：
| 场景 | 展示方式 |
|------|---------|
| 升级 | 完整面板（属性变化用 +N 标注） |
| 获得新技能 | 技能描述 + 效果 |
| 获得装备 | 装备属性 + 与当前装备对比 |
| 战斗中 | 关键数值（伤害、暴击）融入描写 |
| 日常 | 不展示数值 |

### 数值融入叙事

**错误示范**：
> 他使用了【火球术Lv.3】，造成了 1500 点伤害。对方的 HP 从 5000 降到了 3500。

**正确示范**：
> 火球术已经升到三级了。他感受到掌心的温度比以前更炽热，一团拳头大的火球呼啸而出，正中目标胸口。对方踉跄后退，铠甲上烧出一个焦黑的凹痕——这一击至少削去了对方三成的生命力。

### 数值平衡

**避免数值膨胀**：
- 早期：伤害 10-100
- 中期：伤害 100-1000
- 后期：伤害 1000-10000
- ❌ 避免：伤害从 100 膨胀到 100000000

**控制方法**：
- 使用百分比而非绝对数值
- 引入新的伤害类型而非单纯数值增长
- 用质变代替量变（从物理伤害到元素伤害到法则伤害）

## 常见陷阱

### 1. 面板依赖症
❌ 每章都贴完整属性面板
- 解决：只在关键节点展示，日常用叙事代替

### 2. 数值通胀
❌ 数值增长失控，后期动辄百万伤害
- 解决：设计数值上限，用质变代替量变

### 3. 系统万能
❌ 系统能解决所有问题
- 解决：系统有盲区、有 bug、有限制

### 4. 副本流水线
❌ 每个副本都是「打怪 → 打 Boss → 拿奖励」
- 解决：每个副本有独特机制和叙事

### 5. 角色工具化
❌ 队友只是「输出位」「奶位」「坦克位」
- 解决：给每个队友独立的性格和故事线

## 检测清单

### 系统设计检查
- [ ] 属性面板是否简洁明了（5-8 个属性）？
- [ ] 技能是否有明确的使用场景和限制？
- [ ] 装备品质分级是否一致？
- [ ] 数值增长曲线是否合理（无通胀）？

### 叙事检查
- [ ] 数值展示是否融入叙事（非机械罗列）？
- [ ] 副本是否有独特机制和环境叙事？
- [ ] 角色是否有超越「职业定位」的个性？
- [ ] 系统是否有合理的限制和盲区？
